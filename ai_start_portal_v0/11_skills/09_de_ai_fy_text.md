# 09 De-AI-fy Text

## Назначение

Убрать из текста AI-отпечатки: пустые обобщения, рекламный тон, одинаковый ритм, фальшивую человечность, уверенные утверждения без источника и форматирование, похожее на ответ чата.

Важно: это не "детектор AI". Это редакционный чек-лист. Мы не доказываем происхождение текста, мы проверяем, что именно делает его слабым.

## Когда использовать

- текст "звучит как AI написал";
- черновик похож на пресс-релиз без автора;
- AI добавил масштаб, которого нет в источнике;
- текст слишком гладкий, симметричный и уверенный;
- перед выпуском нужен быстрый smell test;
- нужно показать, почему команда "сделай человечнее" не работает.

## Основной prompt

```text
Проверь текст на AI-отпечатки.

Не переписывай сразу.
Не делай вывод "это точно AI".
Сделай редакционную диагностику: что ослабляет текст и что нужно поправить.

Проверь 5 групп признаков.

Группа 1. Пластиковые формулы:
1. "это не X, это Y" / "дело не в X, дело в Y";
2. "важно отметить", "следует помнить", "необходимо учитывать";
3. "и вот почему это важно", "в заключение", "таким образом";
4. риторический финал, который звучит глубоко, но ничего не добавляет.

Группа 2. Пустая значимость:
1. важный, значимый, уникальный, ключевой, богатый, яркий;
2. "играет важную роль", "подчеркивает значимость";
3. "от X до Y" без реального диапазона;
4. перспективы, потенциал, будущее без источника.

Группа 3. Поверхностный анализ:
1. "демонстрирует", "отражает", "подчеркивает" без конкретного действия;
2. выводы о событии без факта, цитаты, числа, места или героя;
3. пересказ вместо объяснения, что именно изменилось.

Группа 4. Машинный ритм и структура:
1. одинаковые абзацы;
2. триады и параллельные конструкции без смысла;
3. много деепричастных оборотов;
4. слишком ровные переходы;
5. злоупотребление длинным тире.

Группа 5. Чатовое оформление:
1. лишний Markdown;
2. жирные заголовки перед каждым абзацем;
3. вертикальные списки там, где нужен связный текст;
4. отдельные разделы "краткое содержание", "вывод", "перспективы" без редакционной задачи.

Для каждого найденного места дай:
- точную цитату;
- тип признака;
- почему это риск для редакционного текста;
- чем заменить: факт / цитата / место / герой / число / действие человека / короткая правка;
- можно ли оставить в этом формате.

После диагностики предложи версию не длиннее исходника.
Новые факты не добавляй.
Если факта нет в источнике, пиши: "нужно проверить".

Текст:
[вставьте текст]
```

## Parallel Critic Mode

Если текст важный, запустить 4 критика.

### Critic A. Generic Detector

Ищет общие фразы, пустую значимость, рекламность, поверхностный анализ.

Выход: цитата, тип проблемы, короткая правка.

### Critic B. Rhythm Analyzer

Ищет одинаковую длину фраз, одинаковые абзацы, триады, слишком ровные переходы, лишние длинные тире.

Исключение: полезные пошаговые списки не трогать.

### Critic C. Specificity Checker

Ищет места, где нужен факт, цитата, место, герой, число, таймкод, документ или наблюдаемая деталь.

### Critic D. Fact / Source Checker

Ищет утверждения, которые требуют проверки:

- даты, должности, имена;
- статистика, проценты, числа;
- причинно-следственные связи;
- названия компаний и продуктов;
- выводы о значимости, перспективах, реакции аудитории.

## Wikipedia AI-Tells Add-On

Русская Википедия описывает типовые признаки сгенерированного текста как совокупность симптомов: чрезмерная важность, позитивная рекламная лексика, поверхностные аналитические глаголы, обтекаемость, дидактические оговорки, машинная структура и чатовая разметка.

Использовать этот список как редакционный фильтр, а не как обвинение.

Команда:

```text
Проверь текст по расширенному списку AI-признаков.
Раздели результат:
1. точно убрать;
2. можно оставить в этом формате;
3. нужно проверить по источнику;
4. нужно спросить редактора.
```

## Rewrite Rules

- итоговая версия не длиннее исходника;
- источник важнее красивой фразы;
- абстракцию заменять на проверяемую конкретику;
- рекламный тон заменять на спокойную редакционную формулировку;
- не добавлять личную эмоцию за автора;
- не добавлять факты ради живости;
- полезные списки, команды, таймкоды и технические фрагменты не ломать;
- финал не должен притворяться моралью.

## Human Gate

- все новые детали должны быть в источнике;
- если факт не подтвержден, он уходит в "нужно проверить";
- автор или редактор решает, где допустима ирония;
- хороший результат может быть менее гладким, но более точным;
- финальная версия читается вслух.
