# TV Style Graph Live Fill

Рабочий лист для демонстрации: заполняется вместе с редакцией в чате или агенте.

Цель: не собрать идеальный Style Graph за один раз, а показать первый живой проход. На выходе должна появиться рабочая рамка продукта, 3-5 первых правил, anti-style и первые decision traces.

## Как Сергей запускает

```text
Мы сейчас собираем не "стиль всей редакции", а TV Style Graph для одного конкретного продукта.

Продукт - это рубрика, формат или регулярная серия:
Telegram-посты о городских событиях, анонсы сюжетов, карточки, подводки, сценарии, спецпроект.

Пожалуйста, отвечайте как редакторы: что уже знаете, что точно нельзя, какие тексты считаете удачными.
Если чего-то нет, пишем "пока нет".
```

## Шаг 1. Выбрать продукт

Сергей спрашивает:

```text
Выберите один продукт для демо.
Не весь Telegram и не вся редакция.

Примеры:
1. короткая новость в Telegram;
2. анонс видеосюжета;
3. объясняющий пост;
4. карточки;
5. подводка ведущего;
6. рубрика про людей;
7. другой регулярный формат.
```

Заполнить:

```text
Product:
Channel:
Rubric / format:
Audience:
Repeating job:
```

Подсказки:

- `Product`: что именно пишем.
- `Channel`: где выходит.
- `Audience`: для кого.
- `Repeating job`: какую работу текст делает каждый раз.

Пример:

```text
Product: короткая новость о городском событии
Channel: Telegram
Rubric / format: городская новость
Audience: жители города
Repeating job: быстро сообщить факт и объяснить, что это меняет для людей
```

## Шаг 2. Забрать то, что редакция уже знает

Сергей спрашивает:

```text
Что редакция уже знает про этот продукт?

1. Есть ли tone of voice, правила рубрики, брендбук, старый промпт?
2. Какие слова и формулировки точно нельзя?
3. Как редакция понимает, что текст получился хорошим?
4. Что обычно приходится править руками?
5. Какие 2-3 текста считаются удачными?
6. Какие 1-2 текста считаются "не нашими"?
```

Заполнить:

```text
Declared ToV / rules:
Known restrictions:
Good text criteria:
Frequent manual edits:
Approved examples:
Rejected examples:
```

## Шаг 3. Вставить удачный текст

Сергей просит один короткий удачный пример. Если есть 2-3 примера, лучше взять несколько.

```text
Approved example 01:
[вставить текст]

Why approved:
[почему редакция считает его хорошим]

Source, if available:
[источник / ссылка / нет]
```

## Шаг 4. Разобрать удачный текст

Команда для чата или агента:

```text
Разбери approved example для TV Style Graph.

Не делай финальный портрет стиля.
Вытащи только наблюдаемые механики.

Сделай:
1. Что делает текст.
2. За счет чего он работает.
3. Что совпадает с заявленным ToV / правилами.
4. Что в тексте есть, но не было описано в правилах.
5. Что можно взять в Style Graph продукта.
6. Где риск имитации.
7. Что нужно проверить на следующих примерах.

В конце дай нумерованный выбор для редакции:
- брать в граф;
- не брать;
- tension;
- проверить дальше.
```

## Шаг 5. Редакция выбирает

Заполнить после обсуждения:

```text
Keep:
Reject:
Tension:
Test next:
```

Пример:

```text
Keep:
- начинать с факта, не с оценки;
- объяснять пользу через действие людей;
- не раздувать масштаб события.

Reject:
- "важный шаг";
- "открывает новые возможности";
- финальная мораль.

Tension:
- хочется живее первый экран, но без кликбейта.

Test next:
- проверить, как это работает в анонсе видео.
```

## Шаг 6. Вставить "не наш" текст

```text
Rejected example 01:
[вставить AI-черновик, пресс-релиз или чужой текст]

Why rejected:
[почему не подходит продукту]
```

Команда:

```text
Разбери rejected example для TV Style Graph.

Найди anti-signals:
1. что звучит не как этот продукт;
2. где пресс-релизность;
3. где AI-пластик;
4. где масштаб усилен без источника;
5. где ритм или структура чужие;
6. какие запреты нужно сохранить.

Не переписывай текст.
Сначала дай список rejected mechanics и спроси редакцию, что точно сохраняем как запрет.
```

Заполнить:

```text
Confirmed anti-signals:
```

## Шаг 7. Собрать первые decision traces

Команда:

```text
Собери decision trace ledger по уже подтвержденным решениям.

Формат:
| Context | Observation | Editor decision | Working hypothesis | Editorial effect | Status |

Статусы:
- test;
- keep;
- reject;
- tension.
```

Заполнить:

```text
| Context | Observation | Editor decision | Working hypothesis | Editorial effect | Status |
|---|---|---|---|---|---|
| approved 01 |  |  |  |  | test |
| rejected 01 |  |  |  |  | reject |
```

## Шаг 8. Первый provisional TV Style Graph

Команда:

```text
Собери provisional TV Style Graph.

Важно:
- не выдавай его как финальный;
- раздели stable / emerging / anti-signals / tensions;
- отдельно покажи, что совпало с заявленным ToV, а что извлеклось из текстов;
- добавь, какие 2-3 материала нужны для следующего обновления.

Формат:
## Product frame
## Declared ToV
## Extracted signals
## Emerging rules
## Anti-signals
## Tensions
## Decision traces
## Next materials needed
```

## Что должно получиться в конце демо

```text
Product frame:

Emerging rules:
1.
2.
3.

Anti-signals:
1.
2.
3.

Decision traces:
| Context | Observation | Editor decision | Working hypothesis | Editorial effect | Status |

Next materials:
1.
2.
3.
```

## Что сказать в конце

```text
Это не финальный стиль редакции.
Это первый TV Style Graph для одного продукта.

Он будет становиться точнее после новых примеров:
удачный текст, слабый текст, AI-черновик, правка редактора.

Чем больше редакция сохраняет решений, тем меньше каждый следующий текст приходится объяснять с нуля.
```
